4.  **사용자 안내**: 카드사 및 카드 번호 마지막 4자리 (알림톡 안내 문구용)

## 5. 보안 가이드라인
- **API Key 관리**: 모든 비밀번호/API Key는 `.env.local`에 보관하며 절대 Git에 노출하지 않음.
- **RLS 철저 적용**: 사용자는 오직 자신의 세션과 구독 정보만 읽을 수 있도록 Supabase Policy 적용.
- **빌링키 보안**: 서버측(Server Actions/Functions)에서만 처리하고 클라이언트로 절대 전달하지 않음.

## 6. 상세 결제 검증 및 예약 확정 가이드

PG사와의 연동 시 데이터 무결성을 위해 아래의 **4단계 검증 절차**를 준수합니다.

### Step 1: 결제 요청 기록 (Pre-Payment)
사용자가 결제 버튼을 누르는 순간, DB에 임시 주문 정보를 생성합니다.
- **Table**: `subscriptions`
- **Status**: `pending` (대기 중)
- **필수 저장**: `order_id` (고유 주문 번호), `amount` (결제할 금액)

### Step 2: 웹훅 수신 및 데이터 대조 (Webhook Verification)
PG사로부터 결제 완료 통보(POST 요청)를 받으면 서버측에서 다음을 확인합니다.
1.  **금액 대조**: PG사가 보낸 `amount`와 우리 DB의 `pending` 주문 `amount`가 일치하는가?
2.  **상태 확인**: PG사 결제 결과가 `success` 또는 `paid`인가?
3.  **중복 방지**: 이미 처리된 `order_id`는 아닌가?

### Step 3: 원자적 예약 확정 (Atomic Update)
검증이 통과되면 다음 작업을 **하나의 트랜잭션**으로 처리합니다.
1.  **구독 상태 변경**: `pending` → `active`
2.  **차기 결제일 설정**: 결제 성공 시점으로부터 계산된 차기 날짜 저장
3.  **보관함 상태 업데이트**: 해당 보관함을 점유 상태로 변경 (`lockers.is_active = false`)

### Step 4: 사용자 최종 피드백
- 모든 처리가 완료되면 사용자에게 '예약 완료' 알림톡 발송.
- 프론트엔드에서는 "예약이 완료되었습니다" 메시지 노출.

## 7. 외주 협업 및 인수인계 명세 (Outsourcing & Delivery)

결제 구현 외주 업체와의 효율적인 협업을 위한 체크리스트입니다.

### 7-1. 업체에 제공해야 할 항목 (Provide to Partner)
- **개발 스택**: Next.js (App Router), Supabase (Auth/DB)
- **결제 관련 주소**:
  - Success URL (성공), Fail URL (실패)
  - **Webhook URL** (서버 간 결제 완료 통보용 API 주소)
- **PG사 정보**: 상점 ID(MID), API Key (전달 시 보안 주의)
- **비즈니스 조건**: 결제 직후 빌링키 저장 및 보관함 점유(`is_active=false`) 상태 즉시 갱신 요구
- **영수증 발급**: 결제 완료 시 PG사에서 제공하는 **영수증 조회 URL(`receipt_url`)**을 결제 내역 테이블에 필수로 저장할 것 (마이페이지에서 영수증 보기 기능 지원용)

### 7-2. 업체로부터 인수받아야 할 항목 (Deliverables)
- **전체 소스 코드**: 결제 컴포넌트, 서버측 검증 및 웹훅 API 코드
- **DB 업데이트 스크립트**: 결제 처리를 위해 추가된 Supabase 테이블/컬럼 SQL
- **환경 변수 가이드**: `.env.local`에 포함되어야 할 PG사 관련 키 리스트
- **테뷰 정보**: Sandbox(테스트) 결제 환경용 아이디 및 비밀번호
- **핵심 데이터 확인**: 알림톡 발송을 위한 `next_payment_date`가 DB의 어느 컬럼에 담기는지 확인서

## 8. 물리 사물함 연동 가이드 (Physical Locker Integration)

결제 데이터와 실제 하드웨어를 연결하는 두 가지 전략입니다.

### 8-1. 소프트웨어 기반 (QR + 비밀번호 지시)
1.  **매핑**: 각 사물함 전면에 `locker_id`와 연결된 고유 QR 부착.
2.  **검증**: 유저가 QR 스캔 시, 현재 활성화된 세션(`usage_sessions`)이 있는지 서버에서 검증.
3.  **인증**: 검증 통과 시 해당 사물함의 **현재 비밀번호** 또는 이용 수칙을 화면에 노출.

### 8-2. 하드웨어 기반 (IoT 자동 제어)
1.  **제품 선정**: 원격 제어용 API를 제공하는 IoT 스마트 락 제품 사용.
2.  **연동**: 결제 완료 웹훅 처리 시, 우리 서버가 사물함 업체 API로 '문 열기' 신호(Signal) 전송.
3.  **로그**: 물리적 문 열림 시점을 `usage_sessions`의 실제 시작 시간으로 기록하여 정확도 향상.

> **작업 포인트**: 사물함 제조사에서 제공하는 **'API 연동 가이드 문서'**를 확보하는 것이 가장 우선적인 과제입니다.
